# ウォッチリストグループ 要件定義

## 1. 概要

- 機能名: ウォッチリストグループ
- 英語名: Watchlist Groups
- 内部名: `watchlist_groups`
- 対象画面: `Myウォッチリスト`

ウォッチリストグループは、お気に入り銘柄を投資テーマ、市場、検討状況などの
ユーザー独自の区分で整理する機能である。グループはフォルダではなく、
Myウォッチリスト画面内のセクションとして扱い、初期展開・session内で開閉可能とする。

## 2. 背景・目的

現行のMyウォッチリストは、お気に入りをカードまたはテーブルで一覧表示し、
絞り込み・並べ替え・Cockpitへの遷移・お気に入り解除を提供している。
銘柄数が増えた場合にも比較候補を見失わないよう、この一覧を
「ユーザー独自の投資候補ボード」として整理できるようにする。

本機能は投資候補の整理だけを担い、Investment Score、Ranking、Forecast、
Research、Risk、通知、売買判断を変更しない。

## 3. 対象ユーザーと主なユースケース

- カスタムユーザーが、お気に入りをテーマや市場ごとに整理する。
- defaultユーザーが、現在のStreamlit session内だけでグループを試す。
- 新しいお気に入りを「未分類」で確認し、後から配置する。
- グループ名や表示順を変えても銘柄配置を維持する。
- グループを削除しても、お気に入り銘柄自体は残す。
- スマートフォンまたはiPadで、select操作により確実に配置を変更する。

## 4. 責務の分離

| データ | 責務 | 保存単位 |
| --- | --- | --- |
| お気に入り | Myウォッチリストに銘柄を出すか | 既存`favorites.json`またはdefault session |
| グループ配置 | お気に入りをどのセクションに出すか | 新規`watchlist_groups.json`またはdefault session |
| snapshot | 価格・スコア等の表示cache | 既存`watchlist_snapshots.json`またはdefault session |

グループ配置はお気に入りの存在を上書きしない。表示対象は常に
「現在のお気に入り集合」と「placement」の積で決定する。

## 5. MVP要件

### 5.1 グループ

- ユーザーは任意名のグループを最大20件作成できる。
- グループ名はtrim後1〜32文字、空白のみと制御文字を禁止する。
- trim後の完全一致を同一ユーザー内の重複として拒否する。
- `group_id`はユーザー入力と無関係な安全なランダムIDとする。
- 改名しても`group_id`と配置は変えない。
- 表示順は永続化し、MVPでは「上へ」「下へ」で変更する。
- 説明は任意で最大200文字とし、作成・編集・グループ見出しで扱う。
- トーンは8種のプリセットから選択し、未指定時は未使用・最少使用を優先して自動割り当てする。
- 自由色入力とアイコンはMVP対象外とする。

表示名の基本制約は既存プロフィール名の1〜32文字、空白のみ不可、
制御文字不可に合わせる。重複名と最大件数は本機能固有の制約である。

### 5.2 未分類

- 未分類は保存された実グループではなく、表示時に合成するシステムセクションとする。
- placementがない、参照先グループが存在しない、または不正なplacementの
  お気に入りを未分類に表示する。
- 名称変更・削除・並べ替えはできず、常に最後に表示する。
- 空の場合も配置編集モードでは移動先として表示する。
- 通常表示で空の未分類は非表示とする。

### 5.3 銘柄配置

- 1銘柄は同時に1グループだけに所属する。
- 新規お気に入りはplacementを作らず、未分類に表示する。
- 専用編集dialog内で、各銘柄の移動先selectからグループまたは未分類を選べる。
- 変更は銘柄単位の「保存」で確定し、選択だけでは永続化しない。
- 未分類を選んだ場合はplacementを削除する。
- symbolは既存お気に入りと同じtrim＋uppercaseで扱う。

### 5.4 お気に入り解除・再登録

MVPでは、お気に入り解除後もplacementを残す。

利点:

- 再登録時に以前のグループへ復帰できる。
- お気に入りと配置の責務分離が明確になる。
- 一時的な解除でユーザー整理を失わない。

欠点:

- 表示されないplacementが蓄積する。
- symbolの意味が将来変わった場合に古い配置が復活し得る。
- 「解除で全情報削除」と誤解される可能性がある。

非表示placementはUI件数に含めず、将来の明示的cleanup・プロフィール削除に
連動した全削除を検討する。再登録時の復帰は受入条件に含める。

### 5.5 グループ削除

- 削除前に確認を表示する。
- 「中の銘柄はお気に入りから削除されず、『未分類』に移動します」と明示する。
- 確定時にグループを削除し、その`group_id`を参照するplacementも削除する。
- お気に入り、snapshot、Decision Trailは変更しない。

### 5.6 表示

- 画面上部に「＋ グループを作成」「配置を編集」「グループ別／すべて」を置く。
- グループ別は各グループと未分類を初期展開し、section単位で開閉できる。
- すべて表示は現行カード／テーブル表示、フィルター、並び順を維持する。
- 現行の絞り込み・並び順を適用した結果をグループごとに分割する。
- 既存Cockpit導線、詳細dialog、判断メモ、お気に入り解除を維持する。

### 5.7 ユーザー分離

- カスタムユーザーは`data/user/profiles/<user_id>/watchlist_groups.json`へ保存する。
- defaultユーザーは既存お気に入りと同様、Streamlit session stateだけに保持する。
- defaultからカスタムユーザーへ自動移行しない。
- ユーザー切替後は前ユーザーのdialog、編集モード、select draftを表示しない。

### 5.8 操作と表示品質

- 配置編集はD&Dだけに依存せず、selectを必須とする。
- 編集モード中だけ配置操作を表示する。
- 主要CTA、編集、肯定、補助、危険操作を色・枠線・文言で区別する。
- 色だけで意味を伝えず、文字ラベルとfocus表示を持たせる。
- モバイルの操作対象は原則44px以上を確保する。

## 6. 非MVP要件

- D&Dによる銘柄移動とグループ並べ替え
- グループ内の手動銘柄順
- 1銘柄の複数グループ所属、タグ型配置
- グループ説明、アクセント色、アイコンの編集
- 折りたたみ
- お気に入り追加時の配置先選択
- 自動分類、AI分類、グループ共有・export/import
- 非表示placementの自動cleanup

## 7. 除外範囲

- お気に入りデータ形式の破壊的変更
- Ranking、Investment Score、Forecast、Risk、Research、通知のロジック変更
- 外部provider呼び出し、AI調査、Decision Reportの自動実行
- 認証・クラウド同期・複数端末同時編集保証
- broker・注文送信

## 8. D&D方針

現行環境はStreamlit 1.38.0で、`st.dialog`と`st.fragment`は利用できるが、
標準のsortable/draggable APIはない。`streamlit-sortables`等の外部componentも
現在の依存に含まれない。

よってMVPではD&Dを採用しない。Phase 5で外部componentの保守性、ライセンス、
タッチ操作、iframe/component境界、onedir EXE同梱、イベント重複、
Playwright再現性を小さなspikeで確認する。採用後もselect fallbackを残し、
編集モード中だけD&Dを有効にする。

## 9. 受入条件

- カスタムユーザー間でgroups/placementsが混ざらない。
- defaultではファイルを作らずsession内だけで動作する。
- グループを作成、改名、並べ替え、削除できる。
- 同名・空白・制御文字・33文字以上・21件目を拒否できる。
- 新規お気に入りと不正参照は未分類に表示される。
- 未分類は最後で、改名・削除できない。
- selectで1銘柄を1グループまたは未分類へ移動できる。
- グループ削除時に銘柄はお気に入りのまま未分類へ移る。
- お気に入り解除後は非表示になり、再登録時に以前の配置へ戻る。
- 「すべて」表示で現行のカード／テーブル、絞り込み、並べ替え、Cockpit導線が保たれる。
- 375、810、1080、1366pxでページ全体の横スクロールがなく、主要操作が可能である。
- 保存失敗・破損時に既存お気に入りを失わず、ユーザーへ復旧可能なエラーを示す。

## 10. 将来拡張

schemaはdescription、tone、group order、placement orderを持つ。
ただし未実装値をUIで暗黙利用せず、schema version migrationを通じて段階導入する。
