## Project Summary

Smart Market AI is a Python-based investment support project that combines market-data ingestion, feature generation, screening, deterministic baseline forecasting, model-informed Investment Score, portfolio/risk checks, Streamlit UI, Decision Report context, and Research RAG evidence workflows.

The product direction is to help users compare symbols, inspect provider-backed data, review model signals, understand warnings, and make better-informed investment decisions without turning the app into automated trading.

Current focus is project maturity improvement rather than feature expansion. Functional spec issues and a manual UX review checklist have been introduced so confusing behavior, unclear role boundaries, and investment-advice-like wording can be reviewed before more advanced features are added.

Current UI maturity work includes Ranking visualization polish: existing ranking results are being presented as screening / comparison candidates with summary cards, top-candidate cards, score and confidence charts, selected-candidate breakdown, and a detailed table, without changing backend ranking or scoring logic.

Ranking visualization maturity now uses condition-aware chart profiles, purpose-specific focus summaries, purpose-specific table leading columns, and row-level sorting reasons / checkpoints. The main scatter chart changes by ranking purpose, uses `上昇気配 x 下降警戒` for multi-factor / upside views, falls back to available existing columns, and treats Evaluation Confidence as a data-reliability / coverage support metric except for data-confidence-focused rankings.

Ranking logic now uses `upside_signal_score` and `downside_signal_score` as the primary forecast-derived ranking/cockpit indicators. Direction signal v2 scores forecast edge, model-by-model direction edge, recent price momentum, and simple trend confirmation after volatility adjustment, so future forecast models contribute by direction and magnitude rather than only up/down counts. Agreement / model spread is treated as a confidence adjustment with a 0.70 floor, not as an additive bonus. Ranking / preview signals use fetched feature-history bars rather than only the selected display period, so short ranking periods do not force every symbol into neutral 50 / 50 fallback. `forecast_agreement_score` remains for compatibility and as a supplementary model-consistency metric; older direction net/label fields may remain in backend contracts for compatibility but are not main ranking/cockpit UI indicators.

Ranking UI readability polish keeps the same layout and logic while shortening candidate-card text, using symbol-first chart labels, compacting the display table to purpose-specific key columns, and separating Decision Report preview from raw Markdown. Ranking Decision Report rows are split into compact candidate memo rows and score-detail rows so Markdown remains readable.

Current UI maturity work also includes Symbol Cockpit polish: the cockpit presents existing analysis results as summary header, KPI cards, price/forecast reading guidance, direction signal cards/details, score breakdown, period evaluation, memo, Research Evidence operation/status + evidence cards, structured Decision Report UI, and lower-priority detail expanders, without changing backend scoring, forecast, Research RAG, or report generation logic.

Current product maturity work includes global UI tone polish: higher-contrast dark navy financial SaaS theme tokens in `ui/styles.py`, role-based text hierarchy tokens, consistent cards, badges, tables, buttons, AI accent surfaces, raised input / expander / container treatments, compact numeric display, and calmer chart explanations are being applied without changing feature behavior or scoring logic.
SMAI mascot assets now have a canonical reference image at `ui/assets/mascot/smai-mascot-reference.webp`; new mascot variants should preserve that character identity. Current title illustrations cover Cockpit, Ranking, and Rebalance page headers, while Symbol Cockpit keeps the cockpit title art and also uses `smai-mascot-cutout.png` as a transparent `SMAI Copilot` / `SMAI Insight` presence asset.

Symbol Cockpit now treats Price & Forecast as the hero chart: Summary and KPI cards come first, then the price/forecast chart, followed by an `上昇気配` / `下降警戒` section, Score Breakdown, review memo, confirmation summary, Research Evidence cards, structured Decision Report, and Developer / Data Details.

Routine development must keep deterministic `mock` / `csv` behavior available while allowing explicit opt-in use of live providers such as Yahoo/yfinance.
The configured default provider remains `mock` for local checks and APIs; the Streamlit Market Data provider selector is live-first with `yahoo` as the first/default option and explicit per-fetch opt-in.
For Research RAG / News RAG, product behavior should prioritize current external sources because freshness is central to the feature. `AI調査を更新` is the standard external-source search action for the selected symbol, currently using TDnet + Yahoo Finance as the first live source set and planned to expand to official IR, EDINET, company sites, provider profiles, and news adapters. The Cockpit Research operation card keeps this as the only always-visible action and uses `AI調査で確認すること` before fetch and `企業リサーチレポートを更新しました` after fetch to summarize business overview, checked source scope, and next confirmation; lower-frequency news-only refresh, CSV, and detail-toggle controls are not part of the main path. Local documents remain important for deterministic tests, demo seeds, user-saved archives, private notes, and fallback, but they are no longer the intended primary product source. Normal checks still use fake adapters / fixtures and must not depend on live network. Fetched source text should not be retained by default; persistent archive should require a separate explicit user action.

Current Research Summary maturity work is local-first readability, not external LLM integration. `CompanyResearchSummary` / `CompanyResearchSummaryBuilder` now acts as the primary company-understanding report layer: it first normalizes `CompanyResearchReport`, RAG evidence, provider profile snippets, TDnet traces, news, and `ResearchBrief` facts into `CompanyResearchEvidence`, then maps them into `企業リサーチサマリー`, `定量情報サマリー`, `IR情報サマリー`, and `最新ニュース・開示サマリー`. Business profile extraction separates main businesses, supporting businesses, products/services, regions, sector/industry, and customer segments without using news titles as business-profile evidence; when product/service details are absent, it can show conservative `補完候補` labels rather than treating them as official facts. The company overview now leads with what is understood from the external profile, using company name plus main/supporting businesses, products/services, and regions before stating missing segment / profit composition checks. Yahoo Finance provider-profile fields such as market cap, enterprise value, revenue, operating income, net income, EPS, PER/PBR/ROE, dividend yield, and employee count are now mapped into `QuantitativeSummary` when present, including common raw camelCase keys such as `marketCap`, `trailingPE`, `priceToBook`, `returnOnEquity`, `dividendYield`, and `fullTimeEmployees`; `Currency: JPY` context is used to format raw numeric money and EPS values as Japanese yen where possible. JPY / ratio / employee display formatting is applied and missing items are shown only for values that are actually absent. Zero values are kept as valid values, while empty / None / NaN-like values remain missing. Provider raw labels such as `Provider Symbol`, `Quote Type`, `Website`, and internal source names are stripped from user-facing company summaries and fixed-question answers, while profile sector/industry labels are shown as Japanese display labels where mapped. TDnet traces can appear both as IR rows and latest-topic disclosure rows, using `unparsed` when only title/URL-level evidence is available; latest news / disclosure summaries now use an `概要 / 企業理解上の意味 / 追加確認` shape, classify keyword-based topics such as forecast revisions, shareholder returns, business reorganization, products/services, and governance, dedupe repeated TDnet/title rows, and state whether the item is an official disclosure or an external news item requiring official IR confirmation. IR/news/metric rows distinguish `found`, `missing`, `unparsed`, and `unverified`; AI読み取りメモ is folded into a detail expander instead of the initial view. AI調査更新 is optimized without reducing source quality: TDnet / Yahoo Finance adapters run through the composite adapter in parallel, identical external payload content is not re-registered or re-chunked in the session store, the Streamlit flow uses a short session TTL cache for repeated external fetches, and the rendered summary bundle is reused when the report/news/external-source inputs are unchanged. A display-only `ResearchBrief` / `ResearchMetric` layer still provides detailed AI整理メモ, reading guide, quantitative cards, business overview, confirmation gaps, next actions, and source cards. `InvestmentInsight` / `InvestmentInsightBuilder` and `InvestmentQuestionSummary` / `InvestmentQuestionSummaryBuilder` remain display-only secondary reading aids and do not affect Research Score, Investment Score, or ranking order. The first `ResearchFactSummary` slice is implemented as a source-backed fact layer between evidence/news/provider payloads and `ResearchBrief`; it surfaces business overview, key businesses, regions / revenue drivers, checked IR / official disclosures / news, major financial metrics, earnings outlook, shareholder return / dividend policy, recent events, positive materials, caution materials, and missing items as user-readable facts rather than acquisition status. Provider raw fields and English snippets are shaped into Japanese confirmation text for the primary view; the operation card now uses company-research wording and `追加確認`, while source cards, Research Score, source confidence labels, and raw/source details stay behind expanders when needed. If rule-based shaping becomes insufficient, a local lightweight LLM summarizer can be considered in a later phase while keeping normal checks deterministic and falling back to the rule-based fact summary.

## Repository Layout

- `backend/app`: FastAPI entrypoints and application wiring
- `backend/core`: shared contracts, settings, and domain errors
- `backend/marketdata`: providers, provider registry/factory, live adapter metadata, feature snapshots
- `backend/forecast`: baseline models, model registry lite, evaluation, forecast consensus
- `backend/screening`: explainable symbol screening score
- `backend/scoring`: Investment Score contract/service
- `backend/reporting`: Decision Report context contracts and deterministic Markdown/manifest helpers
- `backend/risk`: deterministic pre-trade risk checks
- `backend/portfolio`: portfolio valuation, no-solver rebalance proposal, portfolio-to-risk workflow
- `ui`: Streamlit UI for Market Data / Rebalance workflows
- `tests`: deterministic regression tests
- `Documents`: requirements, design, roadmap, operation, UI, and work-log documents
