# ランキング履歴 要件定義

## 1. 概要

- 機能名: ランキング履歴
- 英語名: Ranking History
- 内部名: `ranking_history`
- 対象: SMAI のローカルユーザープロフィールを使う利用者

ランキング画面で明示的に作成した結果をユーザー別に永続保存し、別セッションでも「当時の条件と結果」を再計算せず確認できるようにする。ランキング履歴は投資判断の振り返り材料であり、売買推奨や運用実績ではない。

## 2. 背景と目的

現在のランキング結果は `st.session_state` にあり、セッション終了後に参照できない。条件、取得日、結果を一体のスナップショットとして残し、候補探索の再現性と確認可能性を高める。

## 3. 主なユースケース

1. ランキング作成完了時に、利用中プロフィールへ結果が1件保存される。
2. 利用者がランキング画面上部から履歴一覧を開く。
3. 作成日、データ取得日、種別、対象、条件、候補数、上位銘柄を比較する。
4. 履歴詳細で保存時点の条件と結果を確認する。
5. 重要な履歴をピン留めし、自動削除から保護する。
6. 過去条件をフォームへ復元し、内容確認後に利用者が再ランキングする。

## 4. MVP 要件

### 保存

- 保存トリガーは `ランキング作成` の処理が正常完了した時だけとする。画面再描画、ソート、検索、条件変更だけでは保存しない。
- 一覧用 index と詳細用 snapshot を分離する。
- snapshot は表示に必要な結果を保持し、詳細表示時にランキングを再計算しない。
- `data_as_of`、provider、取得期間、ランキングロジック版、スキーマ版を保持する。
- 同一ユーザー、同一 signature の保存が5分以内にある場合は重複保存しない。
- 通常履歴はユーザーごとに新しい順で30件。超過時は最古の非ピン留め履歴から削除する。
- ピン留め履歴は自動削除しない。明示削除は可能とする。

### ユーザー分離

- カスタムプロフィールは `data/user/profiles/<user_id>/ranking_history/` 配下に保存する。
- `user_id` は `ui.user_data.current_user_id()` / `profile_data_path()` と同じ検証境界を使う。
- 別ユーザーの index や snapshot を列挙、取得、更新、削除できないことをサービス層で保証する。
- `default` は既存方針どおり永続データを持たない。MVPではランキング履歴の保存・一覧を無効化し、プロフィール作成/選択を案内する。セッションだけの「履歴」は提供しない。

### 一覧と詳細

- 一覧はピン留めを上段、通常履歴を下段に分け、各段を新しい順に表示する。
- 一覧項目は作成日、データ取得日、ランキング種別、対象、条件サマリー、候補件数、保存行数、上位銘柄、ピン留め状態。
- 検索はタイトル、メモ、条件、上位銘柄を対象とする。フィルターはピン留め、商品/市場、ランキング種別を対象とする。
- 詳細では検索条件、保存時メタデータ、保存結果、ピン留め/解除、削除、条件復元を提供する。
- 「このランキングは保存時点の結果です。現在の株価・スコアとは異なる場合があります。」を常時表示する。

### 操作

- 履歴結果からコックピットを開く場合は「現在の銘柄を確認」と明示し、保存行の価格やスコアを現在値として渡さない。
- お気に入り状態は snapshot に監査用の保存時値を持てるが、ボタン状態は現在のユーザー別お気に入りを表示する。両者を同一項目として見せない。
- ピン留め済み履歴の削除は確認 UI を必須とする。
- 条件復元はフォームへ値を設定するだけとし、自動で外部取得や再ランキングを実行しない。

## 5. 非 MVP

- 履歴同士の差分比較、推移グラフ、共有、クラウド同期
- タイトル・メモ編集
- 保存上限のユーザー設定
- snapshot への Research 本文や外部取得原文の保存
- 自動スケジュール実行の保存
- CSV 出力（既存 helper を使えるため Phase 7 候補）

## 6. 除外範囲

- ランキング算出、Investment Score、Forecast、Risk、Research Score の変更
- broker / execution 連携
- 既存ランキング画面の大規模な再設計
- 既存ユーザーデータの破壊的移行

## 7. 受入条件

- 再起動後も選択プロフィールの履歴を参照でき、他プロフィールの履歴は見えない。
- 30件制限、ピン留め保護、短時間重複防止が deterministic test で確認できる。
- snapshot 欠損・不正 JSON が1件あっても一覧全体がクラッシュしない。
- history 表示で保存時データと現在データの境界が明示される。
- 通常ランキング、お気に入り、コックピット、Myウォッチリスト、ユーザー切替を壊さない。

## 8. 将来拡張

`schema_version`、`ranking_logic_version`、`universe_version` を保存し、比較、マイグレーション、再現性表示へ拡張する。snapshot は source text を含めず、将来も evidence の識別子・要約・取得時刻に限定する。
